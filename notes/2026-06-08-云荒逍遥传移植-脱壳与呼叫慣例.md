# 移植到云荒逍遥传：脫殼 + FullSentenceCall 呼叫慣例修正

日期：2026-06-08
相關檔案：`Koeicda/api/FullSentenceCall.hpp`、`Koeicda/util/AobScan.hpp`、`tools/mem_dump.py`

## 背景

把本補丁的 `FullSentenceCall` hook 從天龍八部 (`Ekd5.exe`) 移植到云荒逍遥传 (`【云荒逍遥传】.exe`) 時：
天龍八部用寫死 RVA 就找到了 `FullSentenceCall`，云荒卻位址對不上、特徵碼也掃不到。本筆記記錄查因過程與最終修法。

## 發現 1：云荒有加殼，靜態掃不到

云荒 exe 189MB、但 `SizeOfImage` 只有 `0x75D000`（~7.7MB），其餘 180MB 是 overlay。
殼的特徵：

- imports 只剩 `KERNEL32(21)` + `USER32(2)`（原始 IAT 被砍、runtime 才重建）。
- EP 在最後一個 section，stub 是加密的 `E8 00000000 60 ...`（pushad 型）。
- section 名稱被改成編號式（`0\0ext`、`1\0data`…）、每個都標 EXEC、`VirtualSize ≫ RawSize`。
- 本機無 `upx/die/scylla`，且非 UPX，沒有 CLI 靜態脫殼器可用。

**結論**：磁碟上的 bytes 不是真正的遊戲 code，靜態 AOB scan 必然失敗。

### 解法：記憶體 dump（`tools/mem_dump.py`）

本補丁的 `FindAddress` 本來就是 runtime 對 `GetModuleHandle(nullptr)` 的**記憶體**做 AOB scan，掃的是殼已解密好的 code。
所以不需要還原出可執行的 PE，只要：

1. 用 Locale Emulator 以簡體 locale 正常啟動遊戲，跑到任一有畫面的狀態（殼已在 OEP 把 code 解密）。
2. `python tools/mem_dump.py` → 外部 `OpenProcess`+`ReadProcessMemory` dump 主模組（in-memory image 僅 ~7.7MB）。
   - 外部讀取，一般不會觸發殼的反調試。
   - 產出 `*.raw.bin`（offset==RVA，給 AOB scan）與 `*.rebuilt.exe`（section raw=virtual，給 IDA/x32dbg）。
3. 對 dump 跑特徵碼掃描 / 反組譯。

## 發現 2：同一 RVA、不同 codegen

云荒 base `0x400000` 未 ASLR。`FullSentenceCall` 在云荒**就在同一個 RVA `0x2C8B0`**（天龍八部與云荒同引擎），
但前導 bytes 不同，所以舊特徵碼比對失敗：

```
天龍/曹操 @0x2C8B0:  55 8B EC 83 EC 18 | 8B 45 10 89 45 F8 | C7 45 F4 00 00 00 00
云荒      @0x2C8B0:  55 8B EC 83 EC 18 | 33 C0 89 45 F4 89 45 F0 88 45 FE | 8B 45 10 89 45 F8 | ...
```

`FindAddress` 在 hardcoded RVA 先 `SigMatch` 失敗 → 退去全段掃描也掃不到（舊 sig 整個 image 不存在）→ 回 nullptr，這就是「找不到」。

### 解法：多特徵碼（`FindAddressMulti`）

`AobScan.hpp` 新增 `FindAddressMulti(pats, count, rva, &outIndex)`：依序試多組 sig、任一命中即回傳，並回報命中的是第幾組。
`FullSentenceCall.hpp` 改放兩組變體 sig（A=天龍/曹操、B=云荒）。

## 發現 3（真正的崩潰主因）：呼叫慣例不同

換上云荒 sig 後跑出 `Run-Time Check Failure #0 - The value of ESP was not properly saved across a function call`（Debug 的 RTC 才會偵測）。

比對同一 RVA `0x2CAF1` 的 epilogue：

| 遊戲 | epilogue bytes | 慣例 |
|---|---|---|
| 天龍/曹操 | `8B E5 5D C3` | `__cdecl`（caller 清參數）|
| 云荒 | `8B E5 5D C2 10 00` | `__stdcall`（callee `ret 16` 清 4 參數）|

也就是**同一支邏輯函數，兩款被編成不同呼叫慣例**（云荒 prologue 較長、整體位移，連結尾都不同）。
原本一律用 `__cdecl` 呼叫云荒的 `__stdcall` 函數 → 原函數 `ret 16` 已清、我們 call site 又 `add esp,0x10` →
**stack 雙重清理 → ESP 失衡 → RTC #0**。連 hook 函數本身（被遊戲呼叫）的慣例也得跟著對齊，否則遊戲側堆疊損毀。

### 解法：依變體切換慣例

`FullSentenceCall.hpp`：
- 共用轉換抽成 `FullSentenceCall_Resolve()`。
- 兩個 hook 函數：`MyFullSentenceCall_cdecl`（變體 A）、`MyFullSentenceCall_stdcall`（變體 B）；對應兩個 typedef。
- `Install_FullSentenceCall_Hook` 用 `FindAddressMulti` 的 `outIndex` 決定裝哪一個（0=cdecl、1=stdcall）。
- DEBUG log 會印 `variant=1(stdcall)` 方便確認。

## 排除項目

- `PrintfImpl` / `GameTextPrintf`（曹操傳寫死 RVA+sig 的內部函數）在云荒 fixed-RVA 不符、全段也 0 命中 → 回 nullptr、**根本沒被 hook**，與此崩潰無關。
- 「先註解掉 GameTextPrintf 試試」對云荒無效（它在云荒不會執行）。

## 驗證

Debug / Release x86 皆編譯通過；實機跑云荒對話框不再跳 RTC #0、簡轉繁生效。

## 可重用結論

1. 遇到加殼遊戲 → `tools/mem_dump.py` dump 解密後記憶體再分析，不要靜態掃。
2. 同引擎不同遊戲，函數可能**同 RVA 但不同 codegen**（prologue）→ 用 `FindAddressMulti` 多 sig。
3. 更隱蔽的是**同函數不同呼叫慣例**（`ret` vs `ret N`）→ 必須看 epilogue 確認 `__cdecl`/`__stdcall`，hook 函數與 typedef 都要對齊，否則 ESP 失衡。
